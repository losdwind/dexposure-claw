## 一、Definite errors（审稿人称的"硬性内部矛盾"）

### #1 数据划分 split 在 A.5 与 §5/C.2 之间自相矛盾 —— **成立**

证据：

- §5（`4-bench.tex` "Dataset and split"）：train 2020-03–2024-06（约 222 周），
  validation 2024-07–2024-12（约 26 周），test 2025-01–2025-08（约 33 周）。
- App C.2（`C-bench-details.tex:36-38`）：同样是 222 / 26 / 33，与 §5 一致。
- App A.5（`A-fm-notation.tex` "Architecture and training"）：
  "Training uses 104 weekly snapshots (March 2020–January 2024), validation uses
  12 weeks (February–April 2024), and the leaderboard test split is the remaining
  33 weeks."

逐项复核：

1. **104 周对不上日期区间。** 2020-03 到 2024-01 ≈ 46 个月 ≈ 199–200 周，不是 104 周。
   A.5 内部就自相矛盾（日期 vs 周数）。审稿人说的"≈199"正确。
2. **三段加起来 = 149 周、且止于 2024。** 104+12+33 = 149，而 train+val（104+12=116 周）
   到 2024-04 已结束，所谓 "remaining 33 weeks" 若真是"剩余"应是 2024-05 起约 70 周，
   不可能正好是 33 周、更不可能落在 2025。A.5 这里写 "remaining ... (§app:bench-details)"
   是想指回 C.2 的 2025 划分，但与它自己给的 104/12 起讫月份接不上。
3. **潜在泄漏 leakage。** A.5 的验证窗 Feb–Apr 2024 落在 §5 的训练窗（到 2024-06）之内。

**升级（artifact 证据，与第一版初判相反）：这不是 A.5 写错，而是 §5/C.2 写错。**
发布权重 `checkpoints/dexposure-fm-release/config.json` 实测：
`train_weeks=104, val_weeks=12, test_weeks=8, date_range "2020-03-23 to 2025-01-01", seed=42`，
其 val AUPRC `0.978/0.973/0.967/0.967` 与 A.5 报的数**一字不差**——证明论文里 FM 的数字就来自这个
**104 周** checkpoint，A.5 是对的。而代码里 `TRAIN_SPLIT="2020-03~2024-06"`（222 周）
**只用于训练 EvolveGCN(m3)**；FM 是 `fm_predictor.py` 直接 `load` 预训练 ckpt，根本没用 222 周。
所以 §5/C.2 的"222 周训练"对 FM 不成立，"strict no-leakage split"对 FM 也不能直接成立——
因为 FM 的 date_range 一直延到 **2025-01-01**（评测起点），那 104 周到底取自哪段、是否贴近/触碰
2025 评测窗，config 没写明，必须核实。

判定：**真·实验/artifact 问题**（见第 0 节 A 表 #1），不是改 tex 能了结的。
建议二选一：(a) 用 222 周 split 真重训 FM 并重跑下游；(b) 把 §5/C.2 改成 FM 真实的 104 周窗口，
明确起讫，并做一次泄漏审计证明那 104 周与 2025-01~2025-08 不重叠。

### #2 ground-truth stressed-set 大小 |S|≈283 "不可能" —— **部分成立（矛盾是真的，但成因诊断错了）**

审稿人论点：top-5% × 4,300 protocols ≤ 215，单周更少；283 = 周快照数，疑似手误复制。

核验（这条我去查了真实代码与结果文件）：

- `4-bench.tex:44`：π=0.05、W=4，"yielding |S_t^h| ≈ 283 protocols per week"。
- 真实计算结果 `paper/results/latest/b5_decision__m5_fm_rules.json`：
  `stress_pool_size_mean = 283.5`，`n_tickets_mean = 2.0`，`n_weeks_evaluated = 29`。
- 计算逻辑 `paper/experiments/b5_decision_quality.py`：`STRESS_PERCENTILE = 0.05`，
  `cutoff = int(len(drops) * 0.05)`，stressed = 按绝对权重下跌排序的前 5%。

结论修正：

- **283 不是手误、也不是复制周快照数**，它是 `stress_pool_size_mean` 的真实输出（283.5）。
  审稿人推测的"成因"（accidental reuse of the 283 snapshots figure）不成立。
- **进一步实测（与第一版又有更新）：283 完全正确，错的是正文的节点规模描述。**
  我直接读了真实周快照 `data/historical-network_week_2025-07-01.json` 里的 `2025-07-07`：
  **10,706 个节点 / 74,786 条边**。对其中约 5,670 个下跌节点取 top-5% ≈ 283，与
  `stress_pool_size_mean=283.5` 吻合。所以 metric 没错、ckpt 没错、实验结果没错。
- **真正错的是正文那句** "50–500 protocol nodes and 100–2,000 exposure edges per snapshot"
  ——与真实数据差了**两个数量级**（实测 ~1 万节点 / ~7.5 万边）。另外把这 283 个**节点**
  （含 token 端点，因 `_compute_node_total_weight` 把 edge 两端都计入）称作 "protocols"
  也不准确。

也就是说：审稿人"283 不可能"的结论**不成立**（数据支持 283），但他无意中戳中了另一个真问题——
正文把单周节点/边规模写小了两个数量级。

判定：**书写/口径问题**（见第 0 节 B 表 #2），无需重跑。
建议：把节点/边规模改成与数据一致（~1 万节点、~7.5 万边）；说明 stressed 集是在
protocol+token 的全节点集上取 top-5%。顺带一提，stressed 集含 token 节点、而 ticket 只能指向
protocol，这会进一步压低 recall，是和 #8 联动的设计点，值得在正文交代（但不是 bug）。

### #3 Table 1 中 m2、m6 的 F1/Recall 数学上不可能 —— **部分成立（现象属实，但本质是显示/四舍五入，不是数据错误）**

审稿人论点：F1 ≤ 2·Recall。m2（0.0183 vs 2×0.009=0.0180）、m6（0.0241 vs 2×0.012=0.0240）
都越界，故"as printed"不可能；且 m6 若靠 recall 进位救回则逼出 precision≈1.0。

核验：

- 表中数值（`5-results.tex` headline 表）：m2 F1=0.0183 / R=0.009；m6 F1=0.0241 / R=0.012。
- **F1 的定义**（`paper/experiments/aggregate_rejudge.py:74,92`）：
  `macro_f1 = 2 * mean_p * mean_c / (mean_p + mean_c)`，即 F1 就是 mean-precision 与
  mean-recall 的调和平均。
- 既然 F1 由 P、R 调和平均导出，则 **F1 ≤ 2R 在原始数据里恒成立**（代数恒等式）。
  实测 m5：P=0.600、R=0.009634 → F1=0.01896→0.0190，且 2R=0.01927 > F1，完全一致。

结论：

- 表里 "F1 > 2R" 的越界**纯粹是 recall 只印到 3 位小数造成的进位假象**：真实 mean-recall
  对 m2 ≥ 0.00915、对 m6 ≥ 0.01205，印成 0.009 / 0.012 后才看着越界。这不是计算错误，
  数据本身自洽。审稿人 "impossible as printed" 字面成立，但据此推断"有数据错误"不成立。
- **"m6 逼出 precision≈1.0" 这个子论点是夸大的**：precision 取决于 recall 落在进位区间何处。
  若真实 R=0.0123，则 P≈0.59；只有当 R 恰好压在 0.01205 的下界时 P 才趋近 1。所以不是
  "forces precision≈1.0"，而是"precision 在这套印法下不可被验证"。
- 审稿人的**核心建议是对的**：在 Table 1 增印 precision 列（或把 recall 多印一两位），
  三元组 (P,R,F1) 就可自验，越界假象随之消失。

判定：**书写问题**（见第 0 节 B 表 #3）——数据自洽，纯显示精度问题。
建议：headline 表加一列 ticket precision，或脚注说明 F1 = 调和平均(mean_p, mean_c) 且
recall 已四舍五入到 3 位。

### #4 Table 8 "severe" 行被倒置/不自洽 —— **成立**

证据（`A5-A1_isolated_Detail.tex`，severe regime，三行）：

| Gate | τ_data | Safe-mode% | Prec. | FIR | Interventions |
|------|--------|-----------|-------|------|---------------|
| strict | 0.85 | 100 | 0.490 | 0.000 | 0 |
| on | 0.70 | 0 | 0.538 | 0.400 | **25** |
| off | 0.00 | 0 | 0.538 | 0.541 | **24** |

逐项复核：

1. **gate on=25 > gate off=24，方向反了。** 打开 data-health gate 只可能抑制 ticket（safe-mode
   把 ticket 降级为 no-op），不可能比关掉 gate 还多出 1 个 intervention。
2. **两行 safe-mode 都是 0%，却给出不同结果。** safe-mode=0 意味着 gate 根本没触发
   （dh=0.751 > 0.70 阈值），on 与 off 在该 regime 下应当**完全相同**；但 FIR 给了
   0.400 vs 0.541、interventions 25 vs 24。对照 `extreme_feat` regime：on/off 的 FIR
   (0.267/0.267) 与 interventions (5/5) 确实一致——正说明 severe 这一行是录入/复算错误。
   （小提示：`extreme_feat` 的 precision 0.510 vs 0.503 也有微小不一致，复算时一并查。）

**升级（artifact 证据）：tex 没抄错，错在实验输出本身。** 原始
`paper/results/run_20260517_161248_ablation_gpu/run_a1_isolated_results/a1_isolated.json`
里 severe 的 gate_on（entry[10]）与 gate_off（entry[11]）：`safe_mode_pct` 都是 0.0
（gate 实际没触发，两臂本应严格相同），但 `mean_dh` 不同（0.7513402 vs 0.7513024）、
`n_tickets` 不同（83 vs 82）、`n_intervention` 25 vs 24。`mean_dh` 都不一样说明**降级 mask
在两个 arm 各自独立重采样了**，叠加决策 LLM 的 run-to-run 抖动，于是本应相同的两行有了差异、
甚至 on>off。也就是说这不是录入笔误，是消融的两臂没控制随机性。

判定：**真·实验问题**（见第 0 节 A 表 #4）。
建议：重跑 A1 隔离消融，固定 mask 随机种子（理想情况决策也固定/多次取均值），让 safe_mode=0 时
on/off 严格一致；当前 25 vs 24、FIR 0.400 vs 0.541 是随机噪声，不能解读为 gate 效应。

## 二、Misleading framing（数字没错，但结论被夸大）

### #7 "load-bearing" confidence gate 是在没部署的配置上演示的 —— **成立**

证据：

- "load-bearing"（FIR 0.000→0.429）来自 Table 9（`T4-Ablation.tex`）Panel A，baseline 是
  **m5_fm_rules（规则决策）**，不是部署的 LLM 系统。
- 在部署的 LLM 系统上，同一个 gate 几乎不起作用：`5-results.tex` 明写
  "The safety gate (m6→m7) reduces FIR only marginally (0.448→0.437)"。
- abstract（`main.tex:88-94`）把 FIR 归零的功劳正确地记在 **Sonnet 4.6 换模**上
  （"swapping the decision LLM for a tier-below Sonnet 4.6 ... FIR collapses to 0.000"），
  而不是 gate；decision panel 也显示 m7×Sonnet4.6 的 FIR=0.000。

结论：gate 在它**不部署**的规则系统上被证明有效（0→0.429），在它**实际部署**的 LLM 系统上
几乎失效（0.448→0.437）；真正的安全杠杆是换模（Opus4.7→Sonnet4.6）。审稿人的 framing 批评成立。

判定：**书写问题**（见第 0 节 B 表 #7）——所有数字都对，是 framing 把 gate 的作用说大了。
建议：在正文/ablation 处显式说明 "gate 的 load-bearing 证据来自 m5 规则 baseline；在 LLM
决策层 gate 边际作用很小，FIR 主要由 decision-LLM 选择驱动"，避免读者把 gate 当成主因。

### #8 决策指标天然趋近 0、且全程无显著性检验 —— **成立（且我有更硬的证据）**

核验：

- ground-truth 池每周 `stress_pool_size_mean ≈ 283.5`，而决策端 `n_tickets_mean = 2.0`
  （来自 m5 的 b5 json）。recall@k = |tickets∩stressed|/|stressed| 的天花板约 2/283 ≈ 0.7%，
  所以 recall 被**结构性钉在 ~1%**、F1 被钉在 0.01–0.03，任何合理系统都逃不掉。这正是审稿人
  说的 "top-5% ground truth 与少量 ticket 的 action space 不匹配"，而且我用 n_tickets=2 vs
  pool=283 把它量化坐实了。
- headline "+207%"（旧版 +208%）= 0.0076→0.0233，**绝对变化只有 0.016**，样本
  **n=29 周**，全文（Table 1/2/8、附录）**没有任何置信区间或显著性检验**。
- judge 分差如 gate 的 "+0.04"（Opus 下 m7−m6=2.45−2.41）很可能落在噪声内；论文自己也半承认
  （把 Haiku judge 上更大的 +0.21 称作 "partially a ceiling artifact"）。

结论：审稿人成立，这是**测量效度（measurement validity）**层面的核心问题，不是排版问题。

判定：**分析增补，不必重跑**（见第 0 节 C 表 #8）——per-week 序列（`week_recall_at_k` 等）
已存在结果文件里，CI/检验可直接在现有数据上算；但"F1 被结构性钉死"是真实的测量效度问题，
属于"补分析 + 重新 framing"，不是改字。
建议：(1) 对 F1/recall/judge 的 27/29 周序列报 bootstrap CI 或配对检验（paired test across
weeks）；(2) 明确论证：在 |S|≈283 vs ticket≈2 的结构下，为什么 F1 仍是有意义的比较量
（例如改报 recall@K 曲线或 cost-coverage 前沿），而不是单点 F1 的百分比提升。
