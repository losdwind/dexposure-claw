2.1 DeXposure-Agent Architecture (Algorithm 1)
----------------------------------------------------------------------

  G_t --> DataHealth --> FM Predict --> Monitor --> Scenario --> Decision
           |               |              |           |            |
          DH_t         G_hat_{t+h}     Alerts    Ranked losses  Tickets

  Code: dexposure_agent/agent_loop.py :: run_epoch()

Components as implemented:

| Component           | Code file            | Input              | Output                  | Key params        |
|---------------------|----------------------|--------------------|-------------------------|--------------------|
| DataHealth Gate     | data_health.py       | G_t                | DH_t, safe_mode         | tau_data=0.7       |
| FM Predictor        | fm_predictor.py      | G_t, h             | G_hat_{t+h}             | pi_min=0.5 (hybrid)|
| PredGraph Builder   | pred_graph.py        | FM output dict     | GraphSnapshot           | pi_min=0.2         |
| MC Sampler          | pred_graph.py        | FM output dict     | List[GraphSnapshot]     | M=50               |
| Monitor             | monitor.py           | G_hat, baselines   | Alerts, metrics         | z=2.0, L=26w       |
| Scenario Engine     | scenario.py          | G_hat, S1-S5       | ScenarioSummary         | lambda=0.5         |
| Decision            | decision.py          | Alerts, scenarios  | Tickets                 | tau_conf=0.6, K=10 |

2.2 FM Prediction Strategy (Hybrid)
----------------------------------------------------------------------
The FM predictor (fm_predictor.py) uses a hybrid approach:

  1. KEEP all existing edges from G_t
  2. REWEIGHT: new_w = old_w + prob * FM_residual (probability-weighted)
  3. ADD new edges only if FM existence probability >= pi_min (default 0.5)

This preserves network structure while letting FM adjust weights and predict
new connections. Earlier attempts with hard thresholding (pi_min=0.2, 0.5)
removed too many edges and degraded performance.

Checkpoints: checkpoints/dexposure-fm-release/
  - dexposure-fm-h1.pt     (h=1 week,   AUPRC=0.978)
  - dexposure-fm-h4.pt     (h=4 weeks,  AUPRC=0.973)
  - dexposure-fm-h8-h12.pt (h=8-12 weeks, AUPRC=0.967)

Model: GraphPFN encoder (20M params) + LinkScorer + NodeHead

2.3 Monitoring Metrics (Phi)
----------------------------------------------------------------------
Implemented in monitor.py :: compute_metrics():

| Metric ID | Name                        | Computation                   |
|-----------|-----------------------------|-------------------------------|
| M1        | SIS (max PageRank)          | networkx pagerank, take max   |
| M3        | HHI Concentration           | sum(degree_share^2)           |
| M4        | Network Density             | E / (N * (N-1))               |
| M6        | PageRank Concentration      | Gini of PageRank distribution |
| M7        | Gini Weighted Degree        | Gini of weighted out-degree   |

  Not implemented: M2 (Cross-sector Spillover), M5 (Stress-test Loss)
  PageRank uses networkx (numpy-backed) for performance.

2.4 Stress Scenario Library (S1-S5)
----------------------------------------------------------------------
Implemented in scenario.py :: SCENARIO_LIBRARY:

| ID | Name                    | Type      | Shock                                |
|----|-------------------------|-----------|--------------------------------------|
| S1 | Single protocol failure | top_node  | 100% weight removal on top-1 by TVL  |
| S2 | Bridge cluster failure  | category  | 100% removal on category="bridge"    |
| S3 | Stablecoin de-peg       | category  | 50% reduction on category="stablecoin"|

  Note: S2-S4 require meta_df.csv for category labels. Without it,

3.2 B1: Risk Metric Forecasting
----------------------------------------------------------------------
For each h in {1, 4, 8, 12} weeks:
  1. Load all 283 snapshots, pre-compute metrics + PageRank (cached)
  2. For each of 33 test snapshots at time t:
     a. Find ground-truth snapshot at t+h (within +/- 3 days)
     b. Generate predicted graph via predict_graph(method, G_t, h)
     c. Compute metrics on predicted vs actual
  3. Report averages per horizon

| Metric             | Code                                     |
|--------------------|------------------------------------------|
| PageRank MAE       | mean |PR_pred(node) - PR_actual(node)|     |
| HHI MAE            | |M3_pred - M3_actual|                     |
| Density MAE        | |M4_pred - M4_actual|                     |
| Gini MAE           | |M7_pred - M7_actual|                     |
| Rank Correlation   | scipy.stats.spearmanr on PageRank vectors|
| Trend Consistency  | fraction where sign(delta) matches        |

  Key: For persistence (C2), snap_pred IS snap_t (identity check).
  For FM (C0), snap_pred is a NEW object -> fresh metrics computed.

3.3 B2: Early Warning
----------------------------------------------------------------------
For each stress event (Terra/Luna, FTX, SVB) x alert budget (5, 10, 20):
  1. Load pre-window snapshots + build rolling baseline (26w before)
  2. Rank nodes by weighted degree, take top-budget as "alerted"
  3. Ground truth: protocols with >30% weight loss during event window
  4. Compute precision, recall, F1, lead time, stability

| Metric             | Definition                                   |
|--------------------|----------------------------------------------|
| Lead Time (weeks)  | Earliest pre-window week with overlapping alert|
| Precision@budget   | |alerted & stressed| / |alerted|              |
| Recall@budget      | |alerted & stressed| / |stressed|             |
| F1-warning         | 2*P*R / (P+R)                                 |
| Alert Stability    | 1 - (symmetric_diff flips / max_possible)     |

3.4 B3: Uncertainty Calibration
----------------------------------------------------------------------
For 33 test snapshots, horizon=1 (configurable):
  1. Generate FM prediction as "mean"
  2. Generate MC samples: add N(0, 0.1*weight) noise to each edge
  3. For each metric (M1, M3, M4, M6, M7):
     - Build 90% PI from MC quantiles (5th, 95th percentile)
     - Check if actual value falls inside PI

| Metric      | Target | Definition                              |
|-------------|--------|-----------------------------------------|
| ECE         | 0      | Expected calibration error (10 bins)    |
| PI Coverage | 0.90   | Fraction of actuals inside 90% PI       |
| PI Width    | small  | Mean width of prediction intervals      |
| CRPS        | 0      | Energy-form CRPS from MC samples        |

  MC_NOISE_SIGMA = 0.1 (known to be too small; PI coverage ~1-2%)

3.5 B4: Stress Test
----------------------------------------------------------------------
For 5 scenarios x 29 test snapshots, horizon=4:
  1. Apply shock to predicted graph -> compute contagion loss
  2. Apply shock to actual G_{t+4} -> compute ground-truth loss
  3. Compare predicted vs actual

| Metric                | Definition                              |
|-----------------------|-----------------------------------------|
| Loss MAE              | |pred_loss - actual_loss|                |
| Distressed Count MAE  | |pred_count - actual_count|              |
| Propagation Depth MAE | |pred_depth - actual_depth|              |
| Target Overlap@10     | Jaccard(pred_top10, actual_top10)       |

  Uses scenario.apply_shock() and scenario.compute_contagion_loss()

3.6 B5: Decision Quality
----------------------------------------------------------------------
For 29 test weeks (iter_test_with_baselines, 4-week lookahead):
  1. Run full agent pipeline: data_health -> predict -> monitor -> scenario -> decision
  2. Ground truth: protocols with >20% edge-weight loss in next 4 weeks
  3. Compare ticket targets against truly stressed protocols

| Metric                  | Definition                                      |
|-------------------------|-------------------------------------------------|
| Ticket Precision        | |targeted & stressed| / |targeted|               |
| Risk Reduction          | Mean scenario loss delta (simplified)             |
| Target Stability        | Jaccard(targets_t, targets_{t-1}) averaged        |
| Audit Completeness      | |targeted & stressed| / |stressed|               |
| Suppression Rate        | Fraction of weeks with safe_mode=True             |
| False Intervention Rate | Fraction of high-severity tickets on stable nodes |

  STRESS_THRESHOLD = 0.20, STRESS_LOOKAHEAD = 4 weeks

3.7 B6: Robustness
----------------------------------------------------------------------
At fixed horizon=4, for 5 degradation regimes:
  1. Run clean B1-style evaluation as reference
  2. Apply degradation, re-run B1 metrics
  3. Compute relative_degradation = (degraded - clean) / clean

  Same metrics as B1 (PageRank MAE, HHI MAE, etc.)
  Degradation applied to test snapshots, not training data.

========================================================================
7. KNOWN ISSUES
========================================================================

1. B3 PI coverage ~1-2% vs target 90%
   MC_NOISE_SIGMA=0.1 is too small. Need empirical calibration from
   historical prediction errors.

2. B4 S2-S4 perfect scores without meta_df.csv
   Category-based shocks hit zero nodes when categories are "Unknown".
   Fix: ensure meta_df.csv is in data_dir (data_loader now resolves this).

3. B5 audit_completeness near zero
   Ground truth "stressed" set is very large (3000-8000 protocols with >20%
   loss) vs top_k=10 targets. Consider raising STRESS_THRESHOLD to 50%.

4. Trend consistency = 0.0 for persistence
   By definition: persistence predicts no change, so predicted trend is
   always 0. Any non-zero trend in actuals counts as a miss.

5. FM hybrid strategy: C0 ~ C2 on most metrics
   The hybrid keep-all-edges approach preserves structure so well that
   metrics barely change. FM's value shows primarily in trend_consistency.
